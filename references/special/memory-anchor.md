# 长文记忆锚点系统（Memory Anchor）

> 网文动辄百万字，Codex会话有上下文窗口限制。本系统确保跨会话"不失忆"——每次新会话启动时，只需读一个锚点文件即可恢复全书状态。

## 一、设计原理

Codex 每次会话可能被截断/压缩。与其让模型"回忆"百万字，不如维护一个**压缩态快照**——把全书关键状态浓缩到 2000-3000 token，新会话启动时 30 秒内完成上下文恢复。

## 二、核心文件：`记忆锚点.md`

位于项目根目录 `《书名》/记忆锚点.md`，每次写作会话结束时更新。

## 三、更新规则

### 何时更新
- **每写完一章**：更新进度、人物状态、时间锚点
- **每写完一卷**：更新故事引擎、伏笔追踪
- **每次会话结束前**：强制更新所有七段
- **投稿/签约/改稿后**：更新项目地图和进度

### 更新原则
- 只更新变化的部分，不重写全文
- 人物状态用快照格式（一行一个角色），不要叙述
- 伏笔只记未回收的，回收了就从表里删掉
- 上章结尾钩子必须**原文引用**，不要概括——这是续写的关键上下文

## 四、与 Codex 记忆系统的联动

本系统与 Codex 内置记忆（`E:\EXE\Codex\.codex\memories\`）配合使用：

| 用途 | 用什么 |
|---|---|
| 快速恢复全书上下文 | `记忆锚点.md`（项目级，2000-3000 token） |
| 记住用户偏好、决策历史 | Codex 记忆系统（全局级） |
| 记住本次会话内的临时状态 | 本会话上下文窗口 |

**工作流**：
1. 新会话启动 → 先读 `记忆锚点.md`，30秒内建立全书认知
2. 写作中 → 按需读 `设定/` `大纲/` `正文/` 等详细文件
3. 会话结束前 → 更新 `记忆锚点.md`，确保下次会话可承接

## 五、长篇防记忆衰减

百万字长篇的三个防崩机制：

1. **每卷一个锚点快照**：`记忆锚点.md` 只记当前卷，历史卷的快照存在 `存档/卷X_锚点.md`
2. **伏笔编号永不回收**：F001-F999 全书唯一，回收了在表中标 ✅，不要删号——防止后续章引用已回收伏笔
3. **时间锚点用绝对刻度**：写成"Day156"而非"几天后"，写成"第4月"而非"一段时间后"
